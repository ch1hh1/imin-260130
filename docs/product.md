# AIご案内係 — プロジェクト要件定義書

**版：v1.0**  
**ベース文書：AIご案内係1.2（2026-01-18 JST）**

---

## 1. プロジェクト概要

**AIご案内係**は、B2B向けカスタマーサポート／総務・受付向けサービスである。

問い合わせへの回答を効率化し、総務・受付・社内ヘルプデスク等の**一次対応を支援**する。顧客企業固有の規程・マニュアル・FAQ・業務手順を参照可能にし、**属人化の排除**と**応対品質の平準化**を実現する。

---

## 2. 目的・ゴール

| 目的 | 達成内容 |
|------|----------|
| 工数削減 | 問い合わせ一次対応の自動化による対応工数の削減 |
| 品質平準化 | 回答根拠を社内文書に寄せ、応対品質の平準化 |
| 運用定着 | 管理画面による更新により、ナレッジ改善サイクルの定着 |
| 導入リスク低減 | 中小向けに導入障壁を下げ、段階導入で失敗確率を下げる |

---

## 3. 機能要件

### 3.1 ユーザー向け（AIご案内係：フロント）

| No. | 機能 | 要件内容 |
|-----|------|----------|
| F-001 | 質問入力 | テキストによる質問入力が可能であること |
| F-002 | 回答表示 | 回答の表示。必要に応じて参照元・根拠の提示 |
| F-003 | 会話継続 | 追加質問により会話を継続できること |
| F-004 | ガードレール | 禁止トピック・危険質問の抑止、免責表示 |
| F-005 | 利用ログ | 会話履歴を記録し、管理者が閲覧可能であること |

### 3.2 管理者向け（管理画面）

| No. | 機能 | 要件内容 |
|-----|------|----------|
| F-101 | FAQ／ナレッジ登録 | ワード、詳細、カテゴリ、タグ、改定日、版の登録が可能であること |
| F-102 | 承認フロー | 登録→レビュー→公開のフロー（簡易運用モードも可） |
| F-103 | 検索・編集・アーカイブ | ナレッジの検索・編集・アーカイブが可能であること |
| F-104 | 利用状況分析 | 質問件数、一次解決率、未回答率、上位カテゴリの分析表示 |
| F-105 | 権限管理 | 管理者／編集者／閲覧者のロールによる権限制御 |

### 3.3 ナレッジ・ドキュメント管理

| No. | 機能 | 要件内容 |
|-----|------|----------|
| F-201 | ドキュメント投入 | FAQ、規程、マニュアル等の投入が可能であること |
| F-202 | メタデータ付与 | 部署・版・適用範囲等のメタデータ付与 |
| F-203 | 検索品質改善 | インデックス更新等による検索品質の改善 |

## 4. 非機能要件

### 4.1 セキュリティ要件（最低ライン）

| 分類 | 要件 |
|------|------|
| アクセス制御 | RBAC、管理画面MFA（要検討）、IP制限（要検討） |
| 監査 | 操作ログ、ナレッジ改定履歴の記録・保持 |
| データ保護 | TLS、保存データ暗号化（要方式）、バックアップ暗号化（要方式） |
| 情報取り扱い | 個人情報・機微情報の入力ガイド、ログ保持期間／マスキング（要検討） |
| 運用 | 脆弱性対応、インシデント対応フローの整備 |

### 4.2 提供形態・責任分界

- **提供形態**：クラウド提供は行わず、顧客環境内（オンプレ／閉域／ローカル）に限定する。
- **責任分界**：インフラ・運用主体は顧客側、アプリ・設定テンプレート・アップデート・既知不具合修正は提供側とし、契約で明文化する。

### 4.3 参照枠組み

- **OWASP ASVS**：Webアプリのセキュア開発要件・検証基準

## 5. 技術要件・アーキテクチャ

### 5.1 パッケージ構成

| レイヤー | 技術・製品 | 役割 |
|----------|------------|------|
| Webアプリ | Laravel | フロント（AIご案内係）＋管理画面を統合。認証・RBAC・ログ閲覧・ナレッジ更新UI |
| DB | MySQL | FAQ、メタ情報、ログ等 |

### 5.2 処理フロー

1. 利用者がAIご案内係に質問
2. フロントがナレッジ・FAQを検索し回答を取得
3. 回答をフロントに表示
4. 管理画面での更新→ナレッジ更新→精度改善

### 5.3 Dockerパッケージ（インフラ前提）

- 提供物は **docker compose** で起動可能な構成とする。
- 想定コンテナ：laravel-app、mysql、reverse-proxy（必要時）。

---

## 6. 制約・前提条件

- 顧客環境内（オンプレ／閉域／ローカル）でのみ提供し、クラウド提供は行わない。
- 中小・零細企業等を想定し、「はじめてでも導入できる運用設計」を前提とする。

---

## 7. 成果物・評価指標（KPI）

### 7.1 成果物（開発・導入）

- Docker Composeによるパッケージ（Laravel＋DB等）
- 管理画面（ナレッジ登録・承認フロー・分析・権限管理）
- フロント（質問・回答・ガードレール・ログ）
- 導入・運用ドキュメント、責任分界の明文化

### 7.2 評価指標（KPI）

| KPI | 内容 |
|-----|------|
| 一次解決率 | 人手介入なしで完了した問い合わせの比率 |
| 削減工数 | 問い合わせ対応時間の削減量 |
| 未回答率／不適切回答率 | 未回答・不適切回答の割合 |
| ナレッジ更新回数 | 運用が回っていることの指標 |
| 利用継続率（解約率） | 継続利用・解約の状況 |

---

## 8. 段階導入・PoC

- **Step1**：FAQボット（検索＋テンプレ回答）で運用定着。
- **Step2**：ナレッジ範囲の拡大、参照元提示で監査性を担保。
- **Step3**：全社／複数部門への展開＋ログ分析→FAQ追加の改善サイクル。

PoCは期間・対象範囲・ナレッジ量・成功基準を事前に合意し、ログ分析レポート・ナレッジ整備提案・本格導入構成案を成果物とする。
